Canonical operations

Deploy. Mint. Transfer.

These payload shapes are taken from the original BRC-20 documentation. The examples use ORDI, which the source documentation marks as fully minted, so use another ticker for a real test.

Deploy BRC-20
{
  "p": "brc-20",
  "op": "deploy",
  "tick": "ordi",
  "max": "21000000",
  "lim": "1000"
}
Original documentation example. Deploy establishes policy but does not change a holder balance.
Mint BRC-20
{
  "p": "brc-20",
  "op": "mint",
  "tick": "ordi",
  "amt": "1000"
}
A valid mint credits only the first owner of the mint inscription, subject to remaining supply and lim.
Create transferable
{
  "p": "brc-20",
  "op": "transfer",
  "tick": "ordi",
  "amt": "100"
}
This is step one. The transfer settles only when this exact inscription is moved in a later transaction.

Field ledger

Each field has a boundary.

Keep values as strings when encoding protocol quantities. The original rules describe a four-letter ticker, supply and mint quantity limits, and decimal precision that defaults to 18.

FieldOperationOriginal rule
pallLiteral protocol identifier: brc-20.
opalldeploy, mint, or transfer.
tickallFour-letter identifier in the original standard. Ticker comparison is case-insensitive.
maxdeployMaximum supply. The original rules limit maximum supply to unsigned 64-bit range.
limdeploy, optionalPer-mint limit. A mint amount must not exceed it when declared.
decdeploy, optionalDecimal precision. Defaults to 18 and cannot exceed 18.
amtmint / transferRequested amount. A transfer must be no larger than currently available balance when inscribed.
Available balance

The preflight number that matters.

Overall balance can include tokens already committed to transfer inscriptions. Available balance is the amount still eligible for a new valid transfer inscription.

Original documentation model
available balance
= overall balance
- active transferable
  inscription amounts
The original documentation illustrates a holder with 1,000 overall units and a 700-unit transferable as having 300 available units.
OPI indexer excerpt
def check_available_balance(pkScript, tick, amount):
    last_balance = get_last_balance(pkScript, tick)
    available_balance = last_balance["available_balance"]
    if available_balance < amount:
        return False
    return True
Implementation example from the OPI BRC-20 indexer. It reflects the source documentation model.
Reader agreement

Validate the whole event.

BRC-20 balances are protocol interpretation. A builder, wallet, and indexer need to agree on the JSON, inscription ownership, and first-transfer behavior.

  • Canonicalize the ticker for lookup and ensure no earlier valid deployment already owns it.
  • Check max, lim, decimal precision, and remaining supply before creating a mint inscription.
  • Ensure the address receiving a mint or transferable first is the intended BRC-20 owner.
  • Compute available balance at the target inscription order, including any active transferables.
  • Select the exact transfer inscription by satpoint for settlement and keep cardinal fee UTXOs separate.
  • After confirmation, verify both inscription ownership and indexed balance state with the selected BRC-20 reader.
