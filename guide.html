Build path

Keep ownership explicit.

The original documentation repeatedly warns about intermediary inscription services and wallets that rotate addresses. A BRC-20 builder needs to make first ownership visible, not implicit.

Resolve the ticker state.

Normalize the ticker and fetch its earliest valid deployment. Read max, lim, decimals, remaining supply, and current reader rules before you build a mint or transfer.

Pin the receiving owner.

For mint, the first owner of the resulting inscription receives balance. Ensure the inscription is created directly to the address the user controls, not a service forwarding address.

Calculate available, not overall.

For a transfer, subtract every still-active transferable amount from the holder’s overall balance. The new transfer amount must fit within what remains available at its block order.

Inscribe and isolate the transferable.

After a valid transfer inscription is created, treat its satpoint as protected. Do not let normal send or fee selection consume it.

Settle with a second Bitcoin transaction.

Send the exact transfer inscription to the recipient’s Ordinals-compatible address. This first movement applies the receiver balance transition.

Reconcile the final state.

After confirmation, check the transferable’s ownership and balances from the same BRC-20 indexer used to preview the event.

Wallet-facing states

Say what the user owns.

A clear BRC-20 interface should not flatten these quantities into one balance. They determine whether another transfer inscription will be valid.

OVERALL

Indexed holder balance

Total balance attributed by the protocol reader. It may include an amount currently reserved by active transfer inscriptions.

AVAILABLE

Eligible for new transfer

Overall balance after subtracting every valid active transferable. This is the preflight value for a new transfer inscription.

TRANSFERABLE

Locked in an inscription

Amount represented by one valid transfer inscription. It is settled only when that inscription is moved for the first time.

Release gate

Broadcast only when all pass.

Each item is grounded in the original documentation’s semantics or the state model used by the reference indexer implementation.

  • Payload parses as JSON and uses p = brc-20 with exactly the intended operation.
  • Deploy ticker has not been claimed by an earlier valid, case-insensitive deployment.
  • Mint amount honors lim, remaining supply, decimal precision, and the chosen indexer’s integer rules.
  • The mint recipient is the first intended owner of the inscription, with no intermediary service taking custody first.
  • Transfer amount is no larger than available balance at the expected inscription order.
  • The transfer satpoint is excluded from automatic coin selection and the recipient uses an Ordinals-compatible address.
  • The final settlement transaction moves the intended transfer inscription, funded by separate cardinal fee inputs.
  • Post-confirmation inspection verifies both inscription owner and indexer-reported balance transition.
Cancellation behavior. To abandon a valid transferable and restore its amount to available balance, the original documentation instructs the holder to send that transfer inscription to themselves. Do not use an unreviewed wallet action that might instead send it as a fee.
